## Pass 3 — Fetch-and-forget gaps (the bug pattern val described)

### 1. `app/api/admin/av/clients/[client_id]/prep-all/route.ts:153` — **BLOCKER**: Prep step 1 wipes the brief

**Bug pattern:** `saveBriefPayload('av', clientId, patch, ...)` passes only the LLM-suggested-blanks patch (not `{...current, ...patch}`). Since `saveBriefPayload` does a **full column replacement** via `SET brief_payload = CAST(? AS JSON)` (`lib/client/brief_store.ts:141`), every brief field NOT in this run's `patch` is silently erased on each "Prep all" click that writes anything in step 1. Blame puts this code at 2026-06-02 — unchanged since then.

**Why this hasn't burned every client every time:** the snapshot at line 131 of `brief_store.ts` writes a restore point, so val could recover via the brief versions tab — but the live brief is destroyed every Prep run until someone restores.

**Minimal fix:** change line 153 to `await saveBriefPayload('av', clientId, { ...briefPayload, ...patch }, …)`. `briefPayload` is already in scope at line 107.
**Severity:** BLOCKER.

### 2. `app/api/admin/av/clients/[client_id]/account/route.ts:82-103` — **BLOCKER**: AccountInfoEditor drops `company` / `industry` / `contact_name` from the brief

**Bug pattern:** the route writes:
- `clientName` → `clients.client_name` (line 83) — **never to `brief_payload.company`**
- `industry` → `clients.industry` (line 86) — **never to `brief_payload.industry`**
- `contactName` → `client_users.display_name` (line 100) — **never to `brief_payload.contact_name`**

Yet `app/admin/av/clients/[client_id]/page.tsx:204-206` reads `brief_payload.company` and `brief_payload.contact_name` to seed the dossier; `icp_sharpener.ts:148` reads `company_size` from brief; `getBriefForPrompt` does not surface `contact_name` to drafters but the intake editor + preview/intake do read `company` from brief. So val types Skip Krause's name into AccountInfoEditor → it lives on `client_users.display_name`, but the operator brief / preview / intake editor still show blank `contact_name`.

Same logic applies to `industry` (#516 partly added industry awareness but the editor → brief loop is missing) and `company` (every other surface reads `brief_payload.company` first, falling back to `clients.client_name`).

**Minimal fix:** after the `clients.*` and `client_users.*` writes, fetch the brief, merge `{ company: clientName, industry, contact_name: contactName }` (only when each is provided/non-blank) and `saveBriefPayload` blanks-only — same shape as the website save block at lines 144-156.
**Severity:** BLOCKER (val has hit this exact pattern with `contact_name` — explicitly named in the audit prompt's "Likely still broken" list).

### E. Brand-kit non-canonical keys (`logo_url`, `brand_aesthetic`, `brand_typography`)

See Pass 3 finding #5. Drift between what's persisted and what `INTAKE_KEYS` declares.

### F. Aliases tolerated by `extractBriefSeedFromIntake` (`lib/client/intake_brief.ts:80-117`)

The seed extractor accepts canonical key OR documented alias for each field — these are aliases, not drift, since one canonical key is preferred and the alias is the documented fallback. Examples:
- `key_message` OR `market_position`
- `target_audience` OR `ideal_client`
- `message_support` OR `proof_points` OR `press_awards` OR `client_results`
- `audience_insights` OR `client_problems` OR `client_results`

These are **intentional**: the old marketing form used `ideal_client`/`proof_points`/`client_problems`/`founder_story`; the editor uses `target_audience`/`message_support`/`audience_insights`/`why_advertise`. Documented in the comment header. No fix needed — but note the cost: `key_message` and `market_position` share writers in the LLM filler, and the seed extractor can't tell which the operator meant. Surfaced here so val can decide whether to deprecate the aliases.

### G. Voice composition in `app/api/client/intake/route.ts:108-110`

`brand_voice` is composed from `data.brand_traditional + data.brand_friendly + data.brand_pricing` if `brand_voice` isn't set. The three component fields ARE canonical (`intake_fields.ts:75-80`), so they're stored AND a composed `brand_voice` is written too. Both end up in the brief. Minor drift: someone editing `brand_voice` later doesn't update the three components and vice versa.

### H. Channels: `preferred_channels` vs `content_platforms`

Same pattern: `app/api/client/intake/route.ts:114-123` composes `preferred_channels` from `data.content_platforms` (array). Both keys land. Minor.

## Methodology notes

- Every finding cites a `file:line` confirmed by reading the actual source in this run.
- Where I had to choose between "fetch-and-forget" and "drift" as the bug category I used the prompt's pattern: a key the user TYPED or that a scrape FOUND that doesn't reach the canonical source-of-truth = fetch-and-forget; the same logical fact reachable under multiple key spellings = drift.
- I did not start a dev server or run any code. UI tap-target findings are static-read only — they describe what the className says about element sizes, not what a real device shows.
- `getBriefForPrompt` was treated as the single chokepoint for "what the drafters know about this brand." Any field not added there is, by construction, invisible to PR / audit / outreach / distress / weekly-digest prompts even if val typed it five surfaces ago.
