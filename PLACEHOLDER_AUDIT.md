## BLOCKERS

### `app/admin/av/EnrichButton.tsx:374-379` — "Yes, queue outreach" button only fires `alert()`

What it looks like to val: a styled CTA in the post-enrich summary card titled "📨 Send outreach to
these N leads?" with a gold button labeled **"Yes, queue outreach"**. The card sits right under the
"enrich succeeded" payload — same surface, same visual weight as the "Apply" buttons elsewhere in the
hub. Clicking it fires:

```ts
onClick={() => alert('Noted! Email outreach orchestration is queued for the next build session.')}
```

The surrounding text (lines 368-372) is honest about it being a stub ("clicking 'Yes' below tells the
system you want this functionality") — but the button label, color, and placement all read as "do the
thing." val will click expecting outreach to be queued; nothing is recorded, nothing fires, no system
event is logged.

Minimal fix: either (a) wire it to insert a row into a future `outreach_intent_queue` table (or write a
`system_events` row so val sees the intent surfaced) so the click leaves a real trace, or (b) demote to
a neutral disclosure card with no button until the orchestrator ships. Option (b) is one-PR; option (a)
needs schema + a consumer for the queue.

specifically, grep for the form's POST target and confirm it lands somewhere readable.

### `lead_contacts` table — written by /api/admin/av/contacts but never read

`app/api/admin/av/contacts/route.ts:20,115` describes itself as "Inserts a `lead_contacts` row per
leadId (INSERT ... ON DUPLICATE = no-op)..." and does so. But:

```
$ grep -rE "FROM lead_contacts\b|JOIN lead_contacts\b" --include="*.ts" --include="*.tsx"
(no results)
```

Every operator click that adds a contact writes a row that nothing reads. The lead-detail page reads
the `leads` columns (contact_name, contact_title, phone) directly, not via `lead_contacts`. Either:
- (a) `lead_contacts` was the new normalized contact model and the migration off `leads.*` was never
  finished, OR
- (b) the table is a deprecated leftover that the route should stop writing to.

Minimal fix: pick (a) or (b). If (a): build the read path + change every `lead_audit` / outreach drafter
to JOIN `lead_contacts`. If (b): drop the write code in `contacts/route.ts:115` and the schema table.

`lib/feature-flags.ts` declares 6 flags, all 6 are seeded in `schema/003_seed.sql:45`. The HH flag is
read by every `/api/admin/hh/*` route (4 endpoints) so writes are gated. But:

- `app/admin/layout.tsx:7-10` reads `tab_av_enabled` + `tab_ebw_enabled` only — no `tab_hh_enabled`.
- `components/Sidebar.tsx:190` renders `HH_NAV` unconditionally: `{[...HH_NAV, ...(showAv ? AV_NAV : []),
  ...(showEbw ? EBW_NAV : [])]}`.

Result: flipping `tab_hh_enabled = FALSE` hides nothing in the UI. The HH tab + sub-pages stay visible;
clicking them just 403s from the API. Likely intentional in spirit (HH is the always-on default tenant)
but the flag is misleading.

Minimal fix: either (a) pass `showHh` through `Sidebar` from `admin/layout.tsx` and respect it on line
190, or (b) document in code that `tab_hh_enabled` only gates the API.
